## Bash, Shell

- Bash(Bourne Again Shell)는 Unix 및 Linux에서 가장 널리 사용되는 쉘(Shell) 중 하나
- Shell은 운영 체제와 사용자가 상호작용할 수 있도록 해주는 명령어 인터프리터(Command Line Interface, CLI).
- Unix 계열 운영 체제(Linux, macOS 등)에서 다양한 Shell이 존재함.



### 절대경로 (Absolute Path)

- 루트(/) 디렉터리부터 시작하여 전체 경로를 지정하는 방식

### 상대경로 (Relative Path)

- 현재 작업 중인 디렉터리(pwd로 확인 가능)를 기준으로 경로를 지정하는 방식

### Bash와 Shell의 차이

Bash와 Shell은 같은 개념이 아니라, **Bash는 Shell의 한 종류**



### 대표적인 Shell 종류

| **Shell**                     | **설명**                                    |
| ----------------------------- | ------------------------------------------- |
| **sh (Bourne Shell)**         | 기본적인 Unix Shell, 빠르지만 기능이 제한적 |
| **bash (Bourne Again Shell)** | sh의 확장판, 가장 널리 사용됨               |
| **zsh (Z Shell)**             | bash 기반, 자동 완성·플러그인 지원          |
| **csh (C Shell)**             | C언어 스타일 문법을 지원                    |
| **ksh (Korn Shell)**          | sh 기반으로 성능 개선 및 기능 추가          |

### Tag

#### Git Tag의 종류와 현업에서의 활용 방식

Git의 태그(tag) 는 특정 커밋에 이름을 붙여 식별하는 기능. 보통 버전 관리(Release 관리)에 사용



------



#### Git Tag의 종류



| **태그 유형**                   | **설명**                                                     | **특징**                   |
| ------------------------------- | ------------------------------------------------------------ | -------------------------- |
| **Lightweight Tag (경량 태그)** | 단순히 특정 커밋에 이름을 붙이는 태그                        | 브랜치와 유사, 변경 불가능 |
| **Annotated Tag (주석 태그)**   | 태그에 메시지, 작성자, 날짜, GPG 서명 등을 추가할 수 있는 태그 | 정식 릴리스에 사용됨       |

##### 1. Lightweight Tag (경량 태그)

- 단순히 특정 커밋을 가리키는 **라벨 역할**만 함.
- 개인적인 마일스톤(예: “작업 완료 체크”) 등에 사용됨.

```
git tag v1.0.0
```

##### 2. Annotated Tag (주석 태그)

- 주석, 작성자 정보, 날짜, GPG 서명 등을 포함할 수 있는 태그.
- 공식적인 릴리스(Release) 버전 태깅**에 사용됨.

```
git tag -a v1.0.0 -m "첫 번째 정식 릴리스"
```



##### 현업에서 Git Tag를 사용하는 방식

| **버전 형식** | **의미**                            | **예시**      |
| ------------- | ----------------------------------- | ------------- |
| **MAJOR**     | 큰 변경 (이전 버전과 호환되지 않음) | 1.0.0 → 2.0.0 |
| **MINOR**     | 새로운 기능 추가 (호환 가능)        | 1.1.0 → 1.2.0 |
| **PATCH**     | 버그 수정 (기능 변경 없음)          | 1.0.1 → 1.0.2 |



```
git tag -a v2.1.0 -m "새로운 기능 추가"
```
